Backend test task \- approval-service Стр. 1 

**Тестовое задание** 

approval-service: сервис согласования контента 

**Разработайте backend-сервис, который принимает заявки на согласование контента перед публикацией и фиксирует итоговое решение.** 

**Задача** 

В продукте уже есть публикации, сценарии, готовые материалы, пользователи и рабочие пространства. Нужно сделать отдельный сервис для процесса согласования. Внешние сущности передаются как идентификаторы; реализовывать соседние сервисы не нужно. 

**Основные сценарии** 

• создать заявку на согласование; 

• получить список заявок в workspace; 

• получить одну заявку; 

• согласовать, отклонить или отменить заявку. 

**Пример создания заявки** 

`{`   
`"sourceType": "publication",`   
`"sourceId": "pub_123",`   
`"title": "Instagram reel draft", "description": "Needs final approval", "reviewerUserIds": ["usr_1", "usr_2"] }` 

sourceType: publication, scenario, edit или external. sourceId и reviewerUserIds \- внешние идентификаторы. 

**Решения** 

`Approve: { "comment": "Approved" }`   
`Reject: { "reason": "Brand tone is wrong" }`   
`Cancel: { "reason": "Draft was removed" }` 

**Минимальный HTTP API** 

`GET /health`   
`GET /ready` 

`POST /api/v1/workspaces/{workspace_id}/approval-requests`   
`GET /api/v1/workspaces/{workspace_id}/approval-requests`   
`GET /api/v1/workspaces/{workspace_id}/approval-requests/{request_id}` 

`POST /api/v1/workspaces/{workspace_id}/approval-requests/{request_id}/approve`   
`POST /api/v1/workspaces/{workspace_id}/approval-requests/{request_id}/reject`   
`POST /api/v1/workspaces/{workspace_id}/approval-requests/{request_id}/cancel` 

**Ограничения** 

• данные одного workspace не должны быть доступны из другого workspace; 

• повтор одного и того же клиентского запроса не должен создавать дубли; 

• после финального решения заявка не должна переходить в другое финальное состояние; • каждое успешное изменение должно оставлять след, по которому понятно, кто и что изменил; • сервис должен быть готов к последующей интеграции с другими сервисами через события; • в публичные ответы, логи и события не должны попадать секреты, токены, email, storage keys, signed URLs, provider URLs или сырые provider payloads. 

**Auth** 

Для локального запуска достаточно auth-заглушки. Запрос должен содержать workspace, пользователя и список действий. Формат выберите самостоятельно и опишите в README.
